## Co brakuje do Linear/Stripe

### Faza 1: Spójność wizualna (brakujący fundament)

**1.1 Spacing scale**
- `tailwind.config.js` — zdefiniować `spacing` z świadomą skalą (4/8/12/16/24/32/48/64px)
- Zunifikować `px-3.5` / `px-3` / `px-4` → najbliższy token
- Guard: ESLint `no-restricted-spacing` na spacing values > 40px

**1.2 Border radius audit**
- Wszystkie `rounded-*` w kodzie → zmapować do istniejących tokenów (`--radius-sm/md/lg/xl`)
- Usunąć `rounded-[20px]`, `rounded-[22px]` itd. — tylko tokeny

**1.3 Shadow audit**
- Wszystkie `shadow-*` w JSX → zmapować do `--shadow-card/float/nav`
- Usunąć inline `box-shadow:` style z komponentów

**1.4 Dark mode audit**
- Przejrzeć każdy token w `.dark` — czy wszystkie klasy Tailwind (`bg-success`, `text-danger`) działają w obu trybach
- Sprawdzić kontrasty w dark mode (WCAG AA)

### Faza 2: Komponenty (rozszerzenie biblioteki)

**2.1 Brakujące komponenty**
- `Toast` — teraz singleton z globalnym subscribe, ale brak wariantów (success/error/info) jako tokenized
- `Select/Dropdown` — custom select zamiast raw `<select>`
- `Input` — ustandaryzowany input z label/error/hint (teraz każdy plik pisze własny)
- `Toggle/Switch` — zamiast raw `<button>` z bg-primary作为toggle
- `Avatar` — ustandaryzowany (teraz CharacterAvatar + PersonaAvatarButton = 2 implikacje)
- `Tooltip` — brak w ui/
- `Progress` — bar / circular z tokenized kolorami

**2.2 Komponenty do scalenia**
- `.card` CSS class (21 konsumentów) vs `<Card>` (8 wariantów) — nie scalać mechanicznie, ale ustalić kiedy co
- `.btn-primary` CSS vs `<Button>` — CSS buttons = legacy, zdeprecjonować

**2.3 Compound components**
- `Card.Header` / `Card.Body` / `Card.Footer` — zamiast ręcznego layoutu wewnątrz Card
- `Modal.Header` / `Modal.Body` — już istnieje częściowo, dopracować

### Faza 3: Agent-friendly infrastructure

**3.1 Component catalog ( rozszerzenie /dev/design-system)**
- Dodać **prop table** dla każdego komponentu (teraz tylko w DESIGN_SYSTEM.md)
- Dodać **live code preview** — agent widzi kod JSX obok renderu
- Dodać **"Copy import"** — jeden klik = gotowy import

**3.2 Token reference ( auto-generated)**
- Script który generuje `DESIGN_SYSTEM.md` sekcję tokenów z `tailwind.config.js` + `index.css`
- Żeby tokeny nigdy nie były nieaktualne w docs

**3.3 ESLint preset dla agentów**
- `eslint-plugin-vanguard` — shared preset z wszystkimi guardami
- Agent uruchamia `npx eslint --config vanguard` i wie co jest zabronione
- Zawiera: color guard, button guard, typography guard, spacing guard, shadow guard

**3.4 Component usage analytics**
- Script `grep -r` który pokazuje: "Card jest używany w 45 plikach, Button w 96, Badge w 32"
- Tracking adopcji — czy nowe pliki używają komponentów zamiast hardkodu

### Faza 4: Proces i governance

**4.1 Design review checkpoint**
- Przed merge: sprawdzenie czy nowe komponenty używają tokenów
- Checklist w PR template: "Czy użyłem tokenów zamiast hardkodu?"

**4.2 Deprecation pipeline**
- `.btn-primary` CSS → zastąpione przez `<Button>`
- Stare inline styles → tokeny
- Process: oznacz jako `@deprecated` → next sprint usuń

**4.3 Visual regression testing**
- Playwright screenshoty kluczowych ekranów
- Porównanie przed/po zmianach design tokenów
- Zabezpieczenie przed niezamierzonymi zmianami wizualnymi

## Architektura docelowa (10/10)

```
src/
  components/
    ui/          ← 15-20 shared primitives (Button, Card, Input, Select, Modal, Toast, ...)
    shared/      ← 6-8 cross-feature structural (Layout, PageHeader, EmptyState, ...)
    <feature>/   ← domain-specific, używa ui/ + shared/

tailwind.config.js  ← spacing scale, font sizes, colors (SSOT wizualny)
src/index.css       ← CSS custom properties (SSOT dla tokenów)
docs/DESIGN_SYSTEM.md ← reguły dla agentów (auto-generated z config)

eslint.config.js    ← guards: colors, buttons, typography, spacing, shadows
/dev/design-system  ← live gallery z prop tables + code preview
```

**Kluczowy принцип:** agent AI otwiera `DESIGN_SYSTEM.md`, czyta 5 sekcji, i wie:
1. Jakie tokeny istnieją (kolory, typography, spacing, shadows)
2. Jakie komponenty używać (i jakie mają propy)
3. Czego nie pisać (ESLint guards)
4. Gdzie previewować (dev/design-system)
5. Jak dodać nowy komponent (wzorzec z istniejących)
